## 核心技术要点速览

### 怎么保持"7 天免密登录"？

```
登录成功 → 服务端签发两张令牌：
  ① Access Token (JWT, 15 分钟)   → 每次请求带上，过期就找 ② 续
  ② Refresh Token (随机串, 7 天)  → 存在 SecureStore(Native) / HttpOnly Cookie(Web)

用户打开 App：
  → 本地有 Refresh Token？ → POST /api/auth/refresh → 拿新 Access Token → 自动登录
  → 没有或已过期？ → 跳登录页

结果：只要 7 天内打开过 App，用户永远不需要重新输密码。
```

### 验证码怎么投递？

```
找回密码 / 手机绑定 → 服务端生成 6 位数字码 → 存 DB（15 分钟过期）

投递渠道：
  ├── 邮箱 → 阿里云邮件推送（生产）/ Resend（开发）
  └── 手机 → 阿里云短信服务（预留，暂不接入）

限频：同一目标 60 秒限 1 次，24 小时限 5 次
```

### 用户下线后 Agent 怎么跑？

```
核心思路：Agent 是服务端进程，不依赖客户端在线。

用户发布任务 → 任务写入 DB + FSM 状态 = Searching
     ↓
服务端 task_loop（定时轮询 / 事件驱动）：
  → 扫描所有 Searching/Negotiating 状态的任务
  → 执行 L0→L1→L2 匹配流程
  → 匹配成功 → 状态迁移到 Waiting_Human → 推送通知到客户端

用户是否在线完全不影响 Agent 工作。
区别在于：不同订阅等级，Agent 的能力和响应速度不同（详见第六节）。
```

## 二、认证体系设计

### 2.1 Token 架构

采用 **Access Token + Refresh Token** 双 Token 无状态方案：

| Token | 存储位置 | 有效期 | 用途 |
|-------|----------|--------|------|
| Access Token (JWT) | 客户端内存 / AsyncStorage | **15 分钟** | 每次 API 请求携带，验证身份 |
| Refresh Token | HttpOnly Cookie (Web) / SecureStore (Native) | **7 天** | Access Token 过期后静默刷新 |

**"记住登录"实现细节**：

```
App 启动 → AuthProvider 初始化
  ├── 读取本地 Refresh Token
  │     ├── 存在且未过期 → POST /api/auth/refresh → 拿新 token 对 → 自动登录 ✓
  │     ├── 存在但已过期 → 清除本地存储 → 跳登录页
  │     └── 不存在 → 跳登录页
  └── Access Token 在内存中（不持久化）
       → 每次 API 调用前检查 exp
       → 过期前 30 秒自动用 Refresh Token 续期（静默，用户无感知）
```

> Web 端 Refresh Token 存 HttpOnly Cookie（JS 无法读取，防 XSS）；Native 端存 expo-secure-store（加密存储）。

### 2.2 JWT Payload

```jsonc
{
  "sub": "user_id (uuid)",
  "email": "user@example.com",
  "tier": "pro",             // 订阅等级，中间件可快速判断权限
  "iat": 1741852800,
  "exp": 1741853700
}
```

> JWT 里不放敏感信息（密码、手机号）。签名算法用 `HS256`。

### 2.3 refresh_tokens 表（新建）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uuid | PK | 记录标识 |
| `user_id` | uuid | NOT NULL, FK → users | 所属用户 |
| `token_hash` | varchar(255) | NOT NULL, UNIQUE | refresh token 的 SHA-256 哈希 |
| `device_info` | varchar(255) | 可空 | 设备标识（`"iPhone 15 / Expo"`, `"Chrome / Web"`） |
| `expires_at` | timestamptz | NOT NULL | 过期时间 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 创建时间 |
| `revoked` | boolean | NOT NULL, 默认 false | 是否已吊销 |

**索引**：
- `idx_refresh_tokens_user` (user_id) — 查询用户的所有设备
- `idx_refresh_tokens_hash` (token_hash) UNIQUE — 快速查找 token

**Rotation 机制**：每次刷新都换新 refresh token，旧的立即失效。攻击者盗用旧 token 刷新时会发现已吊销 → 全量吊销该用户所有 token → 强制重新登录。

## 三、核心功能流程

### 3.1 注册

```
POST /api/auth/register  { email, password, name }
     ↓
  1. 邮箱格式 + 密码强度校验（≥8 位，含字母+数字）
  2. 查重：email 是否已注册
  3. bcrypt.hash(password, saltRounds=12) → password_hash
  4. INSERT users (subscription_tier = 'free') → 返回 user_id
  5. 生成 access_token + refresh_token
  6. 返回 { user, accessToken, refreshToken }
```

### 3.2 登录

```
POST /api/auth/login  { email, password }
     ↓
  1. 查 users WHERE email = ?
  2. 不存在 → 401 "邮箱或密码错误"（不暴露邮箱是否注册）
  3. status != 'active' → 403 "账号已被暂停"
  4. bcrypt.compare(password, password_hash)
  5. 不匹配 → 401 "邮箱或密码错误"
  6. 更新 last_login_at
  7. 生成 token 对 + 写 refresh_tokens 表（记录 device_info）
  8. 返回 { user, accessToken, refreshToken }
```

### 3.3 Token 刷新（静默续期）

```
POST /api/auth/refresh  { refreshToken }
     ↓
  1. SHA-256(refreshToken) → token_hash
  2. 查 refresh_tokens WHERE token_hash = ? AND revoked = false
  3. 不存在或已过期 → 401（强制重新登录）
  4. 吊销旧 refresh token（revoked = true）
  5. 生成新 token 对 → 写入新 refresh token 记录
  6. 返回 { accessToken, refreshToken }
```

### 3.4 登出

```
POST /api/auth/logout  { refreshToken }
     ↓
  1. 吊销当前 refresh token（revoked = true）
  2. 客户端清除本地存储的 token
```

### 3.5 找回密码

```
POST /api/auth/forgot-password  { email }
     ↓
  1. 查 users WHERE email = ?
  2. 不存在 → 照样返回 200（不暴露邮箱是否注册）
  3. 生成 6 位验证码 + 15 分钟过期
  4. 存入 password_reset_codes 表
  5. 通过邮件服务发送验证码
     ↓
POST /api/auth/reset-password  { email, code, newPassword }
     ↓
  1. 验证 code 是否有效且未过期
  2. bcrypt.hash(newPassword) → 更新 password_hash
  3. 吊销该用户所有 refresh token（强制全设备重新登录）
  4. 标记验证码已使用
```

### 3.6 password_reset_codes 表（新建）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uuid | PK | 记录标识 |
| `user_id` | uuid | NOT NULL, FK → users | 所属用户 |
| `code` | varchar(10) | NOT NULL | 6 位数字验证码 |
| `expires_at` | timestamptz | NOT NULL | 过期时间（+15 分钟） |
| `used` | boolean | NOT NULL, 默认 false | 是否已使用 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 创建时间 |

**限频**：同一邮箱 60 秒限 1 次，24 小时限 5 次。

## 四、个人信息管理

### 4.1 获取个人信息

```
GET /api/user/profile
Authorization: Bearer <access_token>
     ↓
返回：{ userId, email, name, avatarUrl, gender, birthday,
        subscriptionTier, subscriptionExpiresAt, createdAt }
```

> 不返回 `password_hash`、`status` 等内部字段。

### 4.2 修改个人信息

```
PATCH /api/user/profile  { name?, gender?, birthday? }
     ↓
  1. 验证 token → 提取 user_id
  2. 校验各字段格式（name ≤100 字, gender 枚举值）
  3. UPDATE users SET ... WHERE user_id = ? AND status = 'active'
  4. 更新 updated_at
  5. 返回更新后的 profile
```

### 4.3 修改密码（已登录状态）

```
POST /api/user/change-password  { currentPassword, newPassword }
     ↓
  1. bcrypt.compare(currentPassword, password_hash) → 验证旧密码
  2. 校验新密码强度
  3. bcrypt.hash(newPassword) → 更新 password_hash
  4. 可选：吊销其他设备的 refresh token
```

### 4.4 注销账号

```
POST /api/user/deactivate  { password }
     ↓
  1. 验证密码（二次确认）
  2. UPDATE users SET status = 'deleted', updated_at = now()
  3. 吊销所有 refresh token
  4. 不立即物理删除，保留 30 天（用户反悔可恢复）
```

## 五、头像方案

### 5.1 方案选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **阿里云 OSS** | 国内 CDN 快、和 DashScope 同生态 | 需配置 AK/SK | 生产环境推荐 |
| **Cloudflare R2** | 零出口费、全球 CDN | 国内访问不稳定 | 海外用户为主 |
| **本地文件系统** | 零成本、最简单 | 不可扩展、容器重启丢失 | 仅开发环境 |

**推荐：阿里云 OSS + 服务端 STS 签名直传**

### 5.2 上传流程

```
客户端请求上传 → GET /api/upload/avatar-presign
     ↓
BFF: 验证登录态 → 生成 OSS STS 临时凭证 → 返回 { uploadUrl, objectKey, cdnUrl }
     ↓
客户端直传 OSS → PUT uploadUrl（不过 BFF，减轻服务端压力）
     ↓
上传完成 → PATCH /api/user/profile { avatarUrl: cdnUrl }
```

### 5.3 存储规范

```
OSS Bucket: cosoul-assets
  avatars/{user_id}/{timestamp}-{random}.webp           ← 用户头像
  persona-avatars/{persona_id}/{timestamp}-{random}.webp ← 分身头像
```

- DB 只存相对路径（不含域名），前端拼接 CDN 域名展示
- 上传时自动压缩：400×400 / quality 85 / 转 WebP
- 文件类型白名单：`image/jpeg`, `image/png`, `image/webp`，最大 5MB
- STS 凭证：权限最小化，仅限上传到 `avatars/` 目录，15 分钟过期

### 5.4 开发环境降级

不配 OSS 时走本地上传：`POST /api/upload/avatar` (multipart) → 存 `/tmp/uploads/` → Next.js 静态服务。

## 六、订阅等级体系

### 6.1 等级定义

| 等级 | `subscription_tier` | 月费 | 定位 |
|------|---------------------|------|------|
| **免费版** | `free` | ¥0 | 基础体验，吸引用户 |
| **中级版** | `pro` | 待定 | 核心付费用户 |
| **高级版** | `premium` | 待定 | 重度用户 / 商业场景 |

### 6.2 等级权益对比

| 能力维度 | 免费版 (free) | 中级版 (pro) | 高级版 (premium) |
|----------|--------------|--------------|------------------|
| **LLM 模型** | 基础模型（qwen3.5-plus） | 高级模型（qwen3-max） | 顶级模型（kimi-k2.5 等可选） |
| **响应速度** | 排队（低优先级） | 优先队列 | 专属通道（最低延迟） |
| **分身数量** | 最多 3 个 | 最多 10 个 | 无限制 |
| **同时运行任务数** | 1 个 | 5 个 | 20 个 |
| **离线 Agent 后台运行** | ❌ 不支持 | ✅ 基础（每小时扫描 1 次） | ✅ 实时（事件驱动，秒级响应） |
| **Agent 谈判轮次** | 最多 2 轮 | 最多 5 轮 | 最多 10 轮 |
| **个性化服务** | 基础 Intake 提取 | + 记忆摘要回溯 | + 主动推荐 + 偏好学习 |
| **消息推送** | 仅 App 内通知 | + 邮件通知 | + 短信通知 |

### 6.3 离线 Agent 后台运行 — 技术细节

这是订阅等级差异的核心卖点。Agent 作为服务端进程，用户下线后仍可自主执行任务。

```
                        ┌──────────────────────────────────────┐
                        │         task_loop (服务端)            │
                        │                                      │
  免费用户任务 ─────────→│  低优先级队列                         │
                        │  用户在线时才触发 L0→L1→L2            │
                        │  用户下线 → 任务暂停（Paused）        │
                        │                                      │
  Pro 用户任务 ─────────→│  普通队列                             │
                        │  每小时轮询一次 Searching 状态任务     │
                        │  L2 谈判完成 → 推送通知               │
                        │                                      │
  Premium 用户任务 ────→│  高优先级队列 + 事件驱动               │
                        │  新任务入池 → 立即触发匹配             │
                        │  候选人上线/发布新任务 → 实时响应       │
                        │  L2 多轮谈判 → 全自动 → 推送结果       │
                        └──────────────────────────────────────┘
```

**实现要点**：
- `task_loop` 读取任务时检查 `users.subscription_tier` 决定执行策略
- 免费用户：任务在 `Searching` 状态最多存活 24 小时，超时自动 `Timeout`
- Pro 用户：任务存活 7 天，每小时轮询
- Premium 用户：任务存活 30 天，事件驱动实时匹配

### 6.4 等级校验中间件

```typescript
// 伪代码：在需要等级限制的 API 中校验
function requireTier(minTier: "free" | "pro" | "premium") {
  return async (request: Request) => {
    const { userId } = await requireAuth(request);
    const user = await getUserById(userId);

    // 检查订阅是否过期（过期自动降级为 free）
    if (user.subscriptionExpiresAt && user.subscriptionExpiresAt < new Date()) {
      await downgradeToFree(userId);
      user.subscriptionTier = "free";
    }

    const tierRank = { free: 0, pro: 1, premium: 2 };
    if (tierRank[user.subscriptionTier] < tierRank[minTier]) {
      throw new HttpError(403, "需要升级订阅");
    }
  };
}
```

### 6.5 subscriptions 表（新建，记录订阅变更历史）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|

| `user_id` | uuid | NOT NULL, FK → users | 所属用户 |
| `tier` | varchar(20) | NOT NULL | 变更后的等级 |
| `started_at` | timestamptz | NOT NULL | 订阅开始时间 |

## 九、安全措施

### 9.1 密码安全

| 措施 | 说明 |
|------|------|
| bcrypt 哈希 | `saltRounds = 12`，单次哈希约 250ms，暴力破解不划算 |
| 不存明文 | DB 中只存 `password_hash`，泄库不泄密 |
| 登录错误不区分 | "邮箱或密码错误"（不暴露邮箱是否注册） |
| 密码强度校验 | ≥8 位 + 字母 + 数字 |

### 9.2 Token 安全

| 措施 | 说明 |
|------|------|
| Access Token 短命 | 15 分钟过期，被盗窗口小 |
| Refresh Token 轮换 | 每次刷新换新 token，旧的立即失效 |
| Refresh Token 存哈希 | DB 存 SHA-256 哈希，不存原文 |
| 设备绑定 | 记录 device_info，可查看/踢掉特定设备 |

### 9.3 接口安全

| 措施 | 说明 |
|------|------|
| 限频 (Rate Limit) | 登录：5 次/分钟；注册：3 次/小时；忘记密码：5 次/天 |
| CORS | 只允许自家域名 |
| HTTPS | 强制 HTTPS，禁止 HTTP |
| 输入校验 | 所有字段做长度 + 格式校验，防注入 |
