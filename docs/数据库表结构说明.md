# Cosoul.AI — 数据库表结构说明（11 张表）

> **源码定义**：`packages/core/src/db/schema.ts`
> **迁移 SQL**：`drizzle/0000_melted_inertia.sql`
> **最后更新**：2026-03-10

---

## 表关系总览

```
users (1)
 └──< personas (N)                    一个用户拥有多个 AI 分身
       ├── persona_profiles (1:1)     每个分身一份偏好档案
       ├──< tasks (N)                 每个分身下可有多个任务
       │     ├── task_summaries (1:1) 每个任务一份摘要
       │     ├──< task_vectors (N)    每个任务多条向量（activity/vibe/raw/summary）
       │     ├──< handshake_logs (N)  每个任务的握手日志
       │     └──< chat_messages (N)   每个任务关联的聊天消息
       ├──< contacts (N)             每个分身的联系人列表
       ├──< chat_messages (N)        每个分身的全部聊天消息
       └──< memory_summaries (N)     每个分身的记忆摘要

idempotency_keys                     独立表，防重复握手
```

### 外键删除策略

| 策略 | 含义 | 应用场景 |
|------|------|----------|
| `CASCADE` | 父记录删除时，子记录一起删除 | 用户→分身→任务→向量/握手/摘要 等主链路 |
| `SET NULL` | 父记录删除时，子记录外键置 NULL | chat_messages.task_id、memory_summaries.task_id（消息/记忆不因任务删除而丢失） |
| `NO ACTION` | 不级联（如有引用则阻止删除） | contacts.source_task_id（联系人关系不因任务删除而丢失） |

## 1. users — 用户账号

系统中的真人用户，一个用户可以创建多个 AI 分身。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `user_id` | uuid | PK, 自动生成 | 用户唯一标识 |
| `email` | varchar(255) | NOT NULL, UNIQUE | 登录邮箱，全局唯一 |
| `name` | varchar(100) | 可空 | 显示名称 |
| `avatar_url` | text | 可空 | 头像链接 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 注册时间 |

**数据链路**：注册 → 创建用户 → 基于 user_id 创建多个 personas

## 2. personas — AI 分身

每个用户的独立人格实体，是系统中大多数业务操作的主体。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `persona_id` | uuid | PK, 自动生成 | 分身唯一标识 |
| `user_id` | uuid | NOT NULL, FK → users | 所属用户 |
| `name` | varchar(100) | NOT NULL | 分身名称，如"社交达人 Alice" |
| `avatar` | text | 可空 | 分身专属头像 |
| `bio` | text | 可空 | 分身自我介绍 |
| `settings` | jsonb | 默认 `{}` | 分身级别设置 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 创建时间 |
| `updated_at` | timestamptz | NOT NULL, 默认 now() | 最后修改时间 |

**索引**：`idx_personas_user_id` (user_id) — 加速"查询某用户的所有分身"

### settings 字段详解

| Key | 类型 | 说明 |
|-----|------|------|
| `high_match_mode` | boolean | 高度匹配模式 — 热门 Agent 可开启，大幅提高被申请互动的门槛 |
| `language` | string | 分身语言偏好，如 `"zh-CN"` |

**数据链路**：
```
用户创建分身 → personas 表插入
     ↓
前端切换"当前活跃分身" → 后续所有操作（发布任务、消息、联系人）都基于该 persona_id
```

## 3. persona_profiles — 分身偏好档案

每个分身的 User.md 结构化派生，是 Agent 做决策的核心依据。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `persona_id` | uuid | PK, FK → personas | 与分身 1:1 关系 |
| `profile_text` | text | 可空 | User.md 全文（Markdown 格式） |
| `preferences` | jsonb | 默认 `{}` | 结构化偏好（用于 L0 过滤 + Intake 上下文） |
| `updated_at` | timestamptz | NOT NULL, 默认 now() | 最后修改时间 |

### preferences 字段结构（因分身类型而异）

```jsonc
// 社交类分身
{ "age_range": "20-35", "interests": ["聚餐", "户外"], "deal_breakers": ["传销"] }

// 技术类分身
{ "tech_stack": ["TypeScript", "React"], "interaction": "online" }

// 美食类分身
{ "cuisines": ["日料", "川菜"], "budget": "100-300", "vibe": "精致" }
```

**数据链路**：
```
用户编辑"我的"页面 → 更新 persona_profiles
     ↓
Intake 阶段：Agent 读取 profile_text + preferences → 结合用户偏好引导多轮对话
     ↓
L0 阶段：部分 preferences 字段可参与硬过滤条件
     ↓
L2 阶段：Agent 读取 profile_text 作为 CoT 推理背景
```

## 4. tasks — 任务（核心表）

系统中最核心的表，每条记录代表一个用户发布的需求，由 FSM 状态机驱动全生命周期。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `task_id` | uuid | PK, 自动生成 | 任务唯一标识 |
| `persona_id` | uuid | NOT NULL, FK → personas | 发布任务的分身 |
| `status` | varchar(30) | NOT NULL, 默认 `'Drafting'` | FSM 当前状态 |
| `interaction_type` | varchar(20) | NOT NULL, 默认 `'any'` | 交互方式偏好 |
| `current_partner_id` | uuid | 可空 | 当前匹配对方的 task_id |
| `raw_description` | text | 可空 | 用户核心需求精炼描述 |
| `target_activity` | text | 可空 | 具体活动内容 |
| `target_vibe` | text | 可空 | 期望氛围/对方特质 |
| `detailed_plan` | text | 可空 | 完整需求详情 |
| `entered_status_at` | timestamptz | NOT NULL, 默认 now() | 进入当前状态的时间 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 任务创建时间 |
| `updated_at` | timestamptz | NOT NULL, 默认 now() | 最后修改时间 |
| `version` | integer | NOT NULL, 默认 1 | 乐观锁版本号 |
| `pending_sync` | boolean | NOT NULL, 默认 false | 两阶段写入标记 |
| `hidden` | boolean | NOT NULL, 默认 false | 软删除/隐藏标记 |

**索引**：`idx_tasks_persona_status` (persona_id, status) — 加速"查询某分身特定状态的任务"

### 字段详解

#### `status` — FSM 状态机（9 种状态）

```
Drafting      Intake 多轮对话收集需求中
     ↓
Searching     L0 硬过滤 + L1 语义检索进行中
     ↓
Negotiating   L2 Agent ↔ Agent 握手谈判进行中
     ↓
Waiting_Human 匹配成功，等待真人确认
     ├→ Closed     用户满意 → 发送好友申请 → 进入联系人
     ├→ Revising   用户不满意 → 修改需求 → 重新 Searching
     └→ Listening  用户挂起 → 后台持续匹配
          └→ Waiting_Human（新匹配到时回到等待确认）

Timeout       搜索/谈判超时 → 可重试 → Searching
Failed        匹配失败 → 可重试 → Searching
Cancelled     任何状态都可取消（终态）
```

#### `interaction_type` — 交互方式

| 值 | 含义 | L0 过滤作用 |
|----|------|-------------|
| `online` | 仅线上 | 排除 offline 任务 |
| `offline` | 仅线下 | 排除 online 任务 |
| `any` | 都可以 | 不过滤 |

#### `raw_description` — 用户核心需求（≤50 字）

由 Intake 多轮对话中 LLM 从自然语言提取。

| 阶段 | 作用 |
|------|------|
| Intake | LLM 提取："用户核心需求的精炼描述" |
| L1 | 向量化 → `task_vectors` (field=`raw`)，检索权重 **0.30** |
| L2 | 仅本地 Agent 参考，**不发送给对方** |

#### `target_activity` — 具体活动（≤50 字）

| 阶段 | 作用 |
|------|------|
| Intake | LLM 提取："具体活动内容"，用包容性措辞 |
| L0 | **非空检查** — 必须有内容才进入候选池 |
| L1 | 向量化 → `task_vectors` (field=`activity`)，检索权重 **0.35** |
| L2 | **包含在握手 payload** 中，发送给对方 Agent |

#### `target_vibe` — 期望氛围/对方特质（≤50 字）

| 阶段 | 作用 |
|------|------|
| Intake | LLM 提取："期望的氛围/对方特质"，用扩展性措辞 |
| L0 | **非空检查** — 必须有内容才进入候选池 |
| L1 | 向量化 → `task_vectors` (field=`vibe`)，检索权重 **0.35** |
| L2 | **包含在握手 payload** 中，发送给对方 Agent |

#### `detailed_plan` — 完整需求详情（Markdown 格式）

| 阶段 | 作用 |
|------|------|
| Intake | LLM 提取：记录用户提到的所有细节（时间、地点、人数、偏好、雷区），未提及的标记"未限定" |
| L0 | 不参与 |
| L1 | **不做向量化**（信息过于具体，不适合语义匹配） |
| L2 | 仅本地 Agent CoT 推理参考，**不发送给对方**（保护隐私） |

#### 四个内容字段在三层漏斗中的完整流转

```
用户发布需求 → Intake 多轮对话 → LLM 提取 4 个字段 → 写入 task.md → 同步到 PostgreSQL

                            ┌─────────────────────────────────────────────────┐
                            │            三层漏斗匹配流程                      │
                            ├─────────────────────────────────────────────────┤
                            │                                                 │
                            │  L0 硬过滤 (SQL WHERE)                          │
                            │  ├── interaction_type 匹配？                    │
                            │  ├── target_activity 非空？  ← 必须有活动      │
                            │  └── target_vibe 非空？      ← 必须有氛围      │
                            │           ↓ 候选池                              │
                            │                                                 │
                            │  L1 语义检索 (pgvector HNSW)                    │
                            │  ├── activity embedding × 0.35                  │
                            │  ├── vibe embedding     × 0.35                  │
                            │  └── raw embedding      × 0.30                  │
                            │           ↓ Top-K                               │
                            │                                                 │
                            │  L2 沙盒谈判 (Agent ↔ Agent)                    │
                            │  ├── 发送给对方: target_activity + target_vibe  │
                            │  │              + interaction_type               │
                            │  └── 本地参考: detailed_plan + raw_description  │
                            │              （CoT 推理，不外发，保护隐私）       │
                            └─────────────────────────────────────────────────┘
```

> **设计哲学**：`target_activity` 和 `target_vibe` 是"公开名片"（发给对方 Agent 用于谈判），`detailed_plan` 和 `raw_description` 是"内部参谋"（仅本地 Agent 决策用，保护用户隐私细节）。

#### `current_partner_id` — 当前匹配对方

- 存储对方 **task_id**（不是 persona_id）
- 在 Negotiating 和 Waiting_Human 状态下有值
- 匹配关系变化时更新

#### `entered_status_at` — 进入当前状态的时间

- 每次 FSM 状态迁移时更新
- 用于超时检测：如 Searching 超时 → Timeout，Negotiating 超时 → Timeout

#### `version` — 乐观锁版本号

- 每次更新任务时 `version += 1`
- 防止并发写入冲突：更新时 `WHERE version = expected_version`，不匹配则报错重试
- 配合 `pending_sync` 确保双层映射一致性

#### `pending_sync` — 两阶段写入标记

保障 task.md（文件层，真相源）与 PostgreSQL（派生层）的数据一致性：

```
步骤 1: 写入 task.md + 标记 pending_sync = true
          ↓
步骤 2: 同步到 PostgreSQL / SQLite 派生层
          ↓
步骤 3: 同步成功 → 清除 pending_sync = false
```

**崩溃恢复**：如果步骤 2 中途进程崩溃，重启后检测到 `pending_sync = true`，自动重新同步。

#### `hidden` — 软删除标记

| 值 | 含义 |
|----|------|
| `false` | 正常显示在任务列表中 |
| `true` | 从默认列表中隐藏（数据仍在，未物理删除） |

- 用户可隐藏已完成/已取消/不想看的任务
- 默认查询只返回 `hidden = false` 的任务
- 管理员 `list all` 可查看全部任务（含隐藏）
- 有专门的 `setTaskHidden()` 函数，修改时同步派生层并递增 `version`

## 5. task_summaries — 任务摘要

对任务的自然语言概括 + 标签化，用于展示和跨任务检索。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `task_id` | uuid | PK, FK → tasks | 与任务 1:1 关系 |
| `summary_text` | text | 可空 | 任务的自然语言摘要 |
| `tags` | jsonb | 默认 `[]` | 标签数组，用于分类和检索 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 创建时间 |

### tags 字段示例

```json
["剧本杀", "推理", "社交", "线下"]
["AI", "创业", "合伙人", "已完成"]
["日本", "自由行", "关西", "五一"]
```

**数据链路**：
```
Intake 完成 → LLM 生成摘要文本 + 提取标签 → task_summaries 表
     ↓
task_summary.md 文件（真相源）
     ↓
summary_text 可被向量化 → task_vectors (field='summary') → 参与 L1 检索
     ↓
tags 用于前端展示和分类筛选
```

## 6. task_vectors — Embedding 向量索引

存储 DashScope `text-embedding-v4` 生成的向量，是 L1 语义检索的核心数据源。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uuid | PK, 自动生成 | 记录唯一标识 |
| `task_id` | uuid | NOT NULL, FK → tasks | 所属任务 |
| `field` | varchar(30) | NOT NULL | 向量来源字段 |
| `embedding` | vector(1024) | NOT NULL | 1024 维向量 |
| `model` | varchar(100) | 默认 `'text-embedding-v4'` | 生成该向量的模型 |
| `updated_at` | timestamptz | NOT NULL, 默认 now() | 最后更新时间 |

**索引**：
- `idx_task_vectors_task_field` (task_id, field) **UNIQUE** — 每个任务每个字段只有一条向量
- `idx_task_vectors_embedding_hnsw` HNSW (embedding vector_cosine_ops) — 加速余弦距离近邻搜索

### field 字段枚举

| 值 | 来源 | L1 权重 | 说明 |
|----|------|---------|------|
| `activity` | tasks.target_activity | **0.35** | 活动内容向量 |
| `vibe` | tasks.target_vibe | **0.35** | 氛围特质向量 |
| `raw` | tasks.raw_description | **0.30** | 核心需求向量 |
| `summary` | task_summaries.summary_text | — | 摘要向量（辅助检索） |

**数据链路**：
```
Intake 提取字段 → tasks 表保存文本
     ↓
调用 DashScope text-embedding-v4 API → 生成 vector(1024)
     ↓
写入 task_vectors 表（每个字段一行）
     ↓
L1 语义检索：三字段加权余弦距离
     SELECT ... ORDER BY
       (activity_emb <=> query_emb) * 0.35 +
       (vibe_emb <=> query_emb) * 0.35 +
       (raw_emb <=> query_emb) * 0.30
     LIMIT K;
```

> **注意**：`task_vectors` 在 seed 数据中为空（0 条），因为向量化需要调用 DashScope API。实际运行中用户发布需求后才会生成向量。

## 7. contacts — 联系人

分身级别的好友关系，匹配成功后由 Agent 自动创建。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uuid | PK, 自动生成 | 记录唯一标识 |
| `persona_id` | uuid | NOT NULL, FK → personas | 分身（"我"） |
| `friend_persona_id` | uuid | NOT NULL, FK → personas | 好友分身（"对方"） |
| `status` | varchar(20) | NOT NULL, 默认 `'pending'` | 好友状态 |
| `ai_note` | text | 可空 | AI 生成的好友备注（仅自己可见） |
| `source_task_id` | uuid | 可空, FK → tasks | 通过哪个任务匹配成功的 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 创建时间 |

**索引**：
- `idx_contacts_persona` (persona_id) — 查询"我的联系人"
- `idx_contacts_friend` (friend_persona_id) — 查询"谁加了我"

### status 字段枚举

| 值 | 含义 |
|----|------|
| `pending` | 好友申请中，等待对方确认 |
| `accepted` | 已互为好友 |
| `blocked` | 已屏蔽 |

### ai_note — AI 好友备注

由 Agent 在匹配成功时自动生成，描述对方特点和匹配原因。**仅自己可见，对方看不到。**

示例：`"通过「AI 创业合伙」任务匹配。Alice 是全栈工程师，对 AI 和分布式系统有深入了解。"`

**数据链路**：
```
L2 匹配成功 → Waiting_Human → 用户确认满意
     ↓
Closed → 自动创建双向 contacts 记录（A→B 和 B→A 各一条）
     ↓
Agent 生成 ai_note → 写入 contacts
     ↓
消息 Tab → 联系人列表展示好友 + AI 备注
```

> **注意**：好友关系是**分身级别**的，不是用户级别的。Alice 的"社交达人"和"技术宅"有各自独立的联系人列表。

## 8. handshake_logs — 握手日志

记录 L2 阶段 Agent ↔ Agent 自动谈判的完整报文，用于审计和调试。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uuid | PK, 自动生成 | 记录唯一标识 |
| `task_id` | uuid | NOT NULL, FK → tasks | 关联的任务 |
| `direction` | varchar(10) | NOT NULL | 消息方向 |
| `envelope` | jsonb | NOT NULL | 完整握手报文 |
| `timestamp` | timestamptz | NOT NULL, 默认 now() | 记录时间 |

**索引**：`idx_handshake_task` (task_id) — 查询某任务的全部握手历史

### direction 字段枚举

| 值 | 含义 |
|----|------|
| `outbound` | 我方 Agent 发出的消息 |
| `inbound` | 对方 Agent 发来的消息 |

### envelope 报文结构

```jsonc
{
  "protocol_version": "1.0",
  "message_id": "uuid",
  "action": "PROPOSE | COUNTER_PROPOSE | ACCEPT | REJECT",
  "sender_task_id": "uuid",
  "receiver_task_id": "uuid",
  "round": 1,                    // 谈判轮次
  "payload": {
    "proposal": "自然语言谈判内容",
    "target_activity": "...",     // 来自 tasks 表
    "target_vibe": "...",         // 来自 tasks 表
    "match_score": 0.72,          // L1 匹配分数
    "unresolved": ["待确认事项"]  // COUNTER_PROPOSE 时使用
  }
}
```

### 握手协议流程（action 枚举）

```
PROPOSE          我方 Agent 发起匹配提案
     ↓
COUNTER_PROPOSE  对方 Agent 提出修改建议（可多轮）
     ↓
ACCEPT           双方 Agent 达成一致 → 进入 Waiting_Human
     或
REJECT           对方 Agent 拒绝 → 寻找下一个候选
```

**数据链路**：
```
L1 返回 Top-K → L2 开始谈判
     ↓
我方 Agent 发送 PROPOSE → handshake_logs (direction='outbound')
     ↓
对方 Agent 回复 COUNTER_PROPOSE → handshake_logs (direction='inbound')
     ↓
... 多轮谈判 ...
     ↓
ACCEPT → tasks.status 迁移到 Waiting_Human
REJECT → 尝试下一个候选
```

## 9. chat_messages — 聊天消息

覆盖系统中全部四种消息交互模式 + Intake 对话，是消息 Tab 的数据源。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uuid | PK, 自动生成 | 消息唯一标识 |
| `task_id` | uuid | 可空, FK → tasks (SET NULL) | 关联任务（非任务消息为 null） |
| `persona_id` | uuid | NOT NULL, FK → personas | 消息所属分身（用于消息列表查询） |
| `sender_type` | varchar(20) | NOT NULL | 发送者类型 |
| `sender_id` | uuid | NOT NULL | 发送者的 persona_id |
| `content` | text | NOT NULL | 消息内容 |
| `metadata` | jsonb | 默认 `{}` | 扩展元数据 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 发送时间 |

**索引**：
- `idx_chat_persona` (persona_id) — 查询某分身的全部消息
- `idx_chat_task` (task_id) — 查询某任务关联的全部消息

### sender_type 字段枚举

| 值 | 含义 |
|----|------|
| `human` | 真人用户发送 |
| `agent` | AI Agent 自动发送 |

### metadata 字段 — 交互模式标识

| mode 值 | 含义 | sender_type 组合 |
|---------|------|------------------|
| `human-human` | 人 ↔ 人 直接对话 | 双方都是 `human` |
| `agent-agent` | Agent ↔ Agent 自动协商 | 双方都是 `agent` |
| `agent-human` | Agent 主动联系真人 | 发起方 `agent`，回复方 `human` |
| `human-agent` | 真人与对方 Agent 交流 | 发起方 `human`，回复方 `agent` |
| `intake` | Intake 多轮对话（发布需求） | Agent 引导 + 用户回答 |

Intake 模式下 metadata 额外包含 `step` 字段（对话轮次序号）：

```jsonc
{ "mode": "intake", "step": 3 }
```

Agent-Agent 模式下 metadata 额外包含握手信息：

```jsonc
{ "mode": "agent-agent", "round": 2, "action": "COUNTER_PROPOSE" }
```

**数据链路**：
```
┌───────────────────────────────────────────────────────────┐
│ 消息 Tab 切换分身 → 查询 persona_id 下的全部 chat_messages │
│                                                           │
│  ├── task_id 非空 → 任务相关消息（Agent 协商、Intake）     │
│  └── task_id 为空 → 日常聊天（人-人模式）                  │
│                                                           │
│ 前端根据 metadata.mode 渲染不同的消息气泡样式              │
└───────────────────────────────────────────────────────────┘
```

> **设计要点**：`task_id` 使用 `ON DELETE SET NULL` 而非 CASCADE，确保任务删除后历史消息不丢失。

## 10. idempotency_keys — 幂等控制

防止重复握手的分布式幂等锁，TTL 7 天。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `key` | varchar(255) | PK | 幂等键（格式化字符串） |
| `response` | jsonb | 可空 | 首次处理的响应结果 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 创建时间 |

### key 字段格式

```
handshake:{task_id}:propose:{round}
```

示例：`handshake:abc-123:propose:1`

### 工作原理

```
Agent 发起握手
     ↓
检查 idempotency_keys 中是否存在该 key
     ├── 存在 → 直接返回之前的 response（幂等保护，不重复处理）
     └── 不存在 → 处理请求 → 写入 key + response
                              ↓
                  7 天后由 pg_cron 或应用层定期清理
                  (WHERE created_at < NOW() - INTERVAL '7 days')
```

**使用场景**：网络抖动或进程重启导致 Agent 重复发送同一轮握手消息时，通过幂等键确保只处理一次。

## 11. memory_summaries — 记忆摘要

Agent 对历史对话的压缩摘要，用于长期记忆和 RAG 检索。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uuid | PK, 自动生成 | 记录唯一标识 |
| `persona_id` | uuid | NOT NULL, FK → personas | 所属分身 |
| `task_id` | uuid | 可空, FK → tasks (SET NULL) | 关联任务（如有） |
| `summary_text` | text | NOT NULL | 摘要内容 |
| `source_log_id` | varchar(255) | 可空 | 原始对话日志 ID |
| `turn_count` | integer | 可空 | 原始对话轮数 |
| `created_at` | timestamptz | NOT NULL, 默认 now() | 创建时间 |

**索引**：`idx_memory_persona` (persona_id) — 查询某分身的全部记忆

### source_log_id 格式

指向 `.data/<persona_id>/` 下的原始对话日志文件：

```
2026-03-09-chat.md
```

**数据链路**：
```
Agent 与用户/对方 Agent 进行多轮对话
     ↓
对话超过阈值（如 8 轮）→ LLM 压缩生成摘要
     ↓
写入 memory_summaries（保留 source_log_id 用于回溯原文）
     ↓
后续对话中：Agent 通过 RAG 检索相关 memory_summaries
     ↓
注入 LLM 上下文 → Agent "记住"之前的交互历史
```

**作用**：
- **长期记忆**：避免 LLM 上下文窗口溢出，通过摘要保留关键信息
- **RAG 回溯**：当新任务与历史任务相关时，检索相关记忆注入上下文
- **审计**：通过 source_log_id + turn_count 可追溯到原始对话

> **设计要点**：`task_id` 使用 `ON DELETE SET NULL`，确保任务删除后记忆摘要不丢失（记忆属于分身，不属于任务）。

## 附录：Drizzle pgvector 桥接

由于 Drizzle ORM 尚未内置 `vector` 类型，项目通过 `customType` 自定义桥接：

```typescript
// packages/core/src/db/schema.ts
const vector = customType<{
  data: number[];           // TypeScript 侧：number 数组
  driverParam: string;      // PostgreSQL 侧：字符串格式
  config: { dimensions: number };
}>({
  dataType(config) {
    return `vector(${config?.dimensions ?? 1024})`;  // SQL: vector(1024)
  },
  toDriver(value) {
    return `[${value.join(",")}]`;   // JS → PG: [0.1,0.2,...]
  },
  fromDriver(value) {
    // PG → JS: 解析字符串为 number[]
    return value.replace(/^\[|\]$/g, "").split(",").map(Number);
  },
});
```

HNSW 索引由迁移 SQL 手动创建（Drizzle 不直接支持 HNSW 语法）：

```sql
-- drizzle/0000_melted_inertia.sql 最后一行
CREATE INDEX "idx_task_vectors_embedding_hnsw"
  ON "task_vectors" USING hnsw ("embedding" vector_cosine_ops);
```
